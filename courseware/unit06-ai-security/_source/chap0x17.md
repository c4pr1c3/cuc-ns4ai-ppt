# 为什么「AI 赋能」是必修课

---

## 一个事实：安全运营被「告警洪流」淹没

* 一个中型 SOC 每天产生 **数万～数十万** 条告警
* 真阳性占比常 < 1%，分析师 **疲于奔命、滞后响应**
* 人才缺口 + 数据爆炸 = 传统规则与签名 **力不从心**

---

## AI 在安全里到底「赋能」了什么

> 把**重复、海量、需语义理解**的工作，交给 AI；把**判断、决策、问责**，留给人。

* **速度**：秒级分诊 vs 小时级人工
* **广度**：覆盖长尾未知威胁（异常检测）
* **语义**：读懂日志/代码/情报的自然语言含义
* ⚠️ **不等于**：AI 取代安全工程师——它放大工程师，也放大**盲点**

---

## 本课定位（对应能力框架簇 ⑥「赋能」侧）

* <span style="color:#1a73e8">**会用**</span>：能用 AI 工具加速侦察/漏洞分析/报告
* <span style="color:#1a73e8">**会集成**</span>：把一个 AI 检测/分诊组件接进系统（作品 M6）
* <span style="color:#1a73e8">**会评估**</span>：能量化 AI 组件的收益**与局限**（误报/对抗脆弱性）

# AI 赋能安全的全景图

---

## 「数据→模型→决策→行动」四层

```
┌─────────────────────────────────────────────┐
│ 行动 SOAR  ：剧本化响应、隔离、封禁（人在回路）│
├─────────────────────────────────────────────┤
│ 决策 AI-SOC：告警分诊、优先级、调查摘要（LLM） │
├─────────────────────────────────────────────┤
│ 模型 检测  ：ML-IDS / 异常 / UEBA / 代码审计  │
├─────────────────────────────────────────────┤
│ 数据 采集  ：流量 / 日志 / 端点 / 代码 / 情报  │
└─────────────────────────────────────────────┘
```

---

## 两条技术路线

| 路线 | 代表 | 优势 | 短板 |
| :-: | :- | :- | :- |
| **传统 ML** | Isolation Forest、自编码器、随机森林 | 可解释、轻量、稳 | 需特征工程、难语义 |
| **大模型 (LLM)** | GPT/Claude/本地模型 + RAG | 语义强、零样本、可对话 | 幻觉、慢、贵、可被注入 |

> 工程实践多为**混合**：小模型做检测/打分，大模型做分诊/解释/报告。

# 能力一：AI 驱动的检测

---

## 从「误用检测」到「异常检测」

* **误用检测（签名/规则）**：已知坏 → 命中。第九章的 Snort/Suricata 规则即此类
* **异常检测（AI）**：先学「正常」，偏离即告警。能发现**未知**攻击
* 二者**互补**：签名保下限，异常拓上限

---

## 经典算法直觉（七要素·直觉）

* **Isolation Forest**：随机切分空间，**异常点更容易被孤立**（切分次数少）
* **自编码器（Autoencoder）**：压缩再还原，**还原误差大** = 异常
* **聚类（K-Means/DBSCAN）**：离群的小簇 = 可疑

```python{.python .numberLines}
# Isolation Forest 检测异常流量特征（伪代码）
from sklearn.ensemble import IsolationForest
clf = IsolationForest(contamination=0.01, random_state=42)
clf.fit(X_train)            # 只用「正常」样本训练
scores = -clf.score_samples(X_test)   # 分数越高越异常
```

---

## 从异常检测到 UEBA

* **UEBA**（User & Entity Behavior Analytics）：把「异常」具体到**用户/实体**
    * 张三平时 9-18 点从北京登录 → 凌晨 3 点从海外登录 = 高风险
* 输出**风险评分**而非二元告警，便于分诊排序
* 代表：Microsoft Defender for Cloud、Splunk UBA、Elastic ML

---

## 探索过程（七要素·弯路）

1. ❌ 直接把原始日志喂 LLM → token 爆炸、贵、慢
2. ❌ 只用准确率评估 → 误报淹没 SOC（准确率 99% 仍可能不可用）
3. ✅ **特征工程 + 小模型打分 + LLM 解释** 的分层管线
4. ✅ 用 ** Precision/Recall + MTTR + 分析师反馈** 多维度评估

# 能力二：AI-SOC 与告警分诊、SOAR

---

## LLM 在 SOC 的三类用法

* **分诊（Triage）**：给告警打优先级、归类、去重、合并为「事件」
* **调查（Investigate）**：自动拉相关日志、生成时间线与「调查摘要」
* **报告（Report）**：把分析过程写成可交付的事件报告（中英文）

> 代表实践：Microsoft Security Copilot、Google Sec-PaLM/Gemini for Security、Splunk AI Assistant。

---

## 分诊提示词雏形（实战）

```text{.numberLines}
你是 SOC 分析师。下面是 5 条告警，请：
1) 按 MITRE ATT&CK 战术归类
2) 合并同一攻击链的告警为 1 个事件
3) 给出优先级（P0-P3）与一句话依据
4) 标注需要人工确认的不确定项（禁止臆测）
告警 JSON：{{alerts}}
```

⚠️ 关键：**要求模型显式表达不确定性**，避免幻觉变「假阴性」。

---

## SOAR：从「决策」到「行动」

* **SOAR**（Security Orchestration, Automation & Response）：把响应**剧本化**
    * 例：钓鱼邮件 → 提取 IOC → 查威胁情报 → 命中则隔离邮箱 + 封禁 IP
* AI 介入点：**自然语言 → 剧本**、**运行中动态调整**
* **铁律：高风险动作必须「人在回路」（Human-in-the-Loop）**

# 能力三：AI 辅助渗透与代码审计

---

## 攻防一体：AI 也让「攻击」变快

* **侦察**：LLM 解读 nmap/子域名结果、生成攻击面假设
* **漏洞利用**：辅助分析反编译/反汇编、生成 payload 变体（绕 WAF）
* ⚠️ **伦理与法律边界**：仅限授权范围；本课**不教**编写恶意代码

---

## AI 代码审计（七要素·实战）

* 思路：把**代码 + 安全规范**喂模型，让它指出疑似漏洞并给出修复
* 工具：GitHub Copilot Autofix、Semgrep AI、CodeQL + LLM、GPT-Reviewer

```bash{.bash .numberLines}
# 用 AI 审计一个路由文件（示意）
semgrep --config p/python src/app/ --json > findings.json
# 再用 LLM 对 findings 做去重、误报过滤与修复建议
```

---

## 探索过程（弯路）

1. ❌ 「让 AI 直接找 0day」→ 幻觉严重，产出大量假阳性
2. ❌ 全自动合并修复 → 引入新漏洞或破坏逻辑
3. ✅ **静态分析器（高精度低召回）+ LLM（高召回低精度）** 交叉验证
4. ✅ 修复建议**必须人工 review + 测试**

# 能力四：AI 威胁情报与知识问答（RAG）

---

## RAG = 给 LLM 装上「最新情报大脑」

* LLM 训练有截止日期 → 对**新 CVE/新组织**一无所知
* **RAG**（Retrieval-Augmented Generation）：检索情报库 → 拼进提示 → 生成
* 让「威胁情报问答」可回答：「Log4Shell 怎么检测？用的什么 ATT&CK 技术？」

```
用户问题 → 向量检索(情报库) → Top-K 片段 → LLM 生成(带引用)
```

---

## 安全价值与风险（双向）

* ✅ **赋能**：分析师用自然语言查情报、生成检测规则
* ⚠️ **作为对象（接 Ch15/16）**：RAG 库若被投毒 → 模型输出**被污染的处置建议**
* 本课在 **作品 M6** 同时演示这两面

# AI↔AI 攻防演进

---

## 攻防都在用 AI：螺旋升级

* **防守**：AI 检测、AI 分诊、AI 取证
* **攻击**：AI 生成钓鱼/深度伪造、AI 改造 payload、AI 自动化侦察
* → **AI 防御 AI**：用 AI 检测 AI 生成的攻击（如深度伪造检测、AI 钓鱼识别）

---

## 给工程师的三条判断

1. AI **降低门槛**：新手也能快速出报告 → 但**深度判断仍需人**
2. AI **引入新攻击面**：赋能组件本身可被绕过/投毒（见 Ch15-16）
3. 评估 AI 安全产品：看 **误报率、可解释性、对抗鲁棒性、人在回路**，而非 Demo 效果

# 实战：作品 M6 集成一个 AI 检测组件

---

## 目标（对应能力簇 ⑥·赋能侧·L2）

在你的靶场 Web 应用里**集成一个 AI 驱动的检测/分诊组件**，并度量其效能。

---

## 最小可行实现

```python{.python .numberLines}
# 用 Isolation Forest 对登录行为做异常打分（作品 M6 赋能侧）
from sklearn.ensemble import IsolationForest
import numpy as np

# 特征：[小时, 是否国内, 请求频率, 失败次数]
X = load_login_features()
model = IsolationForest(contamination=0.02).fit(X)
risk = -model.score_samples(X)            # 风险分
alerts = np.where(risk > np.quantile(risk, 0.99))[0]
```

---

## 交付与度量（量规要点）

* 集成进应用日志管线，**实时**输出风险分
* 报告：**Precision/Recall**、误报分析、与纯规则版的对比
* 明确写出 **AI 组件的局限**（哪些攻击它检测不到、为何）
* 详见 [`capstone/m6-ai.md`](capstone/m6-ai.md.v4.html)

# 常见误区与延伸

---

## 常见误区（七要素·误区）

* ❌ 「AI 检测准确率 99%，可以裁掉分析师」→ 1% 误报 × 海量数据 = 灾难
* ❌ 「LLM 不会出错」→ 幻觉会变成**假阴性/假阳性**，必须可解释可纠偏
* ❌ 「买了 AI 安全产品就安全了」→ 赋能组件自身是**新攻击面**
* ❌ 「自动响应全自动」→ 高风险动作**必须人在回路**

---

## 总结与延伸

* AI 赋能 = **分层管线**（小模型检测 + LLM 分诊 + SOAR 行动 + 人在回路）
* 四大能力：检测 / SOC / 渗透审计 / 情报问答
* 关键不是「用不用 AI」，而是**在哪一层用、如何评估、如何兜底**
* 延伸：阅读 MITRE ATLAS（AI 系统攻击知识库）、OWASP LLM Top 10、各厂商 Security Copilot 技术博客
